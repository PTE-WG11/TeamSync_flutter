然后系统要根据不同的角色来分配不同的功能和权限，系统有四种用户：超级管理员，团队管理员，团队成员，访客（未被邀请为团队成员的），超级管理员是系统管理员（系统所有权限）；团队管理员：查看项目进度、项目中的计划完成情况，项目的创建（创建时候分配人员），项目中任务的分配，项目状态的修改，成员的管理等；团队成员：查看分配给自己的项目，在项目中查看任务属于自己的任务下可以创建子任务，子任务来细化这个项目，和项目一样分为待处理，处理中，已完成的项目

补充1

1. 【组织架构】整个公司就是一个大团队，所有人都在这个团队里
2. 【访客权限】只能看到登录页，等待被邀请（无操作权限）
3. 【项目归档】支持归档，不再出现在活跃列表，允许超级管理员硬删除防止测试数据垃圾
   4.【任务分配】 一个任务只能分配个一个成员，一个项目分为多个成员，子任务由成员自己来创建（目的是方便成员自己来进行任务的拆分和计划），任务包含：当前状态（规划中，进行中，未完成，已完成），正常标识（正常，已逾期），任务级别（紧急程度（紧急/高/中/低）），开始时间，结束时间，创建时间，附件，负责人；子任务包含，当前状态，任务级别，正常标识，描述，开始时间，结束时间，创建时间，附件等；状态由各负责人更新自己的部分
4. 【层级深度】子任务层级限制3层 即，任务-子任务-子任务 三层
5. 【数据范围】团队管理员查看"项目进度"时：能看到项目中所有任务的详情，包括成员自己创建的任务
6. 【甘特图依赖】不需要支持依赖关系，就显示看到就可以了，管理员看到所有成员的甘特图，成员看到自己子任务下的甘特图；甘特图默认展示的时间范围是日视图
7. 【实时协作】需要实时通知使用WebSocket；任务被分配时实时推送，任务状态变更时通知相关人员；通知存储：通知是否需要持久化到数据库（方便查看历史通知）
   9.【任务创建权限】只能由团队管理员创建并分配；子任务只能由该任务的负责人自己创建
   任务状态流转：允许任意状态变更（如已完成→进行中），但记录变更历史用于审计
8. 团队中是多个团队管理员的权限相同都可以查看该团队项目中的所有数据

11.现在里面设计的是不是：  项目下有主任务，主任务下有子任务，项目由团队管理员创建，创建后在项目中添加主任务；然后一个团队有多个项目，每个项目下划分为多个主任务和子任务
12.附件存储：采用MinIO（兼容S3的对象存储），支持私有化部署，优于本地文件系统
13.现在先不设计页面结构，这个需要单独设计

1. 【成员的项目可见范围】能看到所有项目（包括不含自己任务的项目），但点进去只能看到分配给自己的任务，其他任务显示为"无权限查看"
2. 【子任务的编辑权限边界】团队管理员能看到成员创建的子任务详情，仅查看成员创建的子任务，子任务完全由负责人自治

3. 【项目成员管理】后期通过"项目设置-成员管理"动态添加/移除
4. 【访客邀请机制】团队管理员在"成员管理"中输入访客在系统中的账户邀请，访客自动成为"团队成员";这里的访客指注册的用户但是没有进入团队的称之为访客，没有注册进不了系统
5. 【任务逾期与提醒】需要定时任务（如每天早9点）推送"今日截止任务"提醒；视觉标记+每日推送，不自动改状态（避免误标）。截至之后就把正常标识改为已逾期；
6. 团队协作层面都是显示主任务的状态，涉及到个人才显示子任务及子任务视图

1. 关于"团队协作层面显示主任务状态"，团队视图（管理员看的项目进度页）：甘特图/看板中只显示主任务节点，要展开子任务，个人视图：成员查看自己的任务时，显示该主任务下的子任务树形结构
主要是统计项目完成进度时，只统计主任务的完成情况，子任务仅作为个人工作分解
2. 子任务是否默认继承父主任务的负责人，且不可更改
3. 正常标识保持"已逾期"、保留逾期记录
4. "无权限查看"的具体表现：显示任务标题但内容脱敏（仅标题可见，详情点击提示无权限）

5. 访客邀请流程；访客是先注册账号（此时为游客，然后登录进去显示等待团队邀请）
6.创建项目时必须至少分配一个成员（不能为空项目）
7. 子任务层级计数 项目 →  主任务 → 子任务 → 子子任务（主任务算第1层，最大深度3）
